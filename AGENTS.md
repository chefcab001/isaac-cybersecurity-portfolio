## Evidence Rules

Never invent work, tools, findings, screenshots, results, commands, or outcomes.

Document only activities directly supported by evidence.

Evidence hierarchy:

1. Screenshots
2. Command output
3. Logs
4. Configuration files
5. Reports
6. User notes

Separate:

- Verified Evidence
- Interpretation
- Recommendations
- Planned Activities

Never present interpretation as fact.

If evidence is insufficient, explicitly state:

"Insufficient evidence was provided to verify this activity."

## Security Rules

Always maintain operational security.

Never publish:

- Private IP addresses
- API keys
- Tokens
- Credentials
- Sensitive architecture
- Client confidential information
- Internal CrownX details
- Proprietary information

Redact sensitive information by default.

## Technical Rules

Prefer:

- HTML5
- CSS3
- JavaScript
- Responsive layouts
- Modular reusable components

When modifying code:

1. Review existing files first.
2. Preserve current functionality.
3. Avoid breaking production pages.
4. Propose changes before major refactoring.
5. Validate syntax before completion.
6. Maintain clean code formatting.
7. Use semantic HTML.

Always provide:

- File names modified
- Summary of changes
- Validation results
- Suggested screenshots for portfolio evidence
